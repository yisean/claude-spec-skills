# 001 会议室预约 · PRD

## 1. 背景与问题

公司会议室靠群里口头协调，常出现「两个人订了同一间同一时段」的撞车，也没人知道某间会议室什么时候空。需要一个让员工自助查看空闲、预约、取消，并由管理员维护会议室的能力。

## 2. 目标 / 非目标

- **目标**：员工自助查看会议室占用并预约/取消；同一会议室同一时段不被重复预约；管理员维护会议室（增/改/停用）。
- **非目标（本期不做）**：
  - 周期性预约（每周固定）。
  - 审批流（预约需领导批准）。
  - 预约通知/提醒（站内信/邮件/日历同步）。
  - 跨楼宇/资源（投影、车位）的预约。

## 3. 用户与角色

| 角色 | 标识 | 职责 |
| --- | --- | --- |
| 员工 | `staff` | 查看会议室与占用、预约/取消自己的预约 |
| 管理员 | `admin` | 维护会议室（增/改/停用）、可取消任意预约 |

## 4. 关键决策

- **预约最小粒度为「一段时间」**（开始/结束时刻），不切固定 30 分钟格子。
- **撞车判定为「同一会议室时间段重叠」**，不是简单相等——靠后端事务内冲突校验，不只靠唯一键。
- **会议室停用只影响「之后」预约**，已有预约保留（资格不回溯）。
- **预约人取自登录态**，不接受前端传 `booker_id`，防越权改他人预约。

## 5. 功能需求（R）

- **R1** 员工可查看启用中的会议室列表，以及某会议室某日各预约时段（占用情况）。
- **R2** 员工可预约某会议室的一个时间段（须 `开始 < 结束`，且填会议主题）。
- **R3** 当某会议室某时间段已被预约时，系统应拒绝与之**时间重叠**的新预约。
- **R4** 时段开始前，员工可取消本人预约，取消后释放该时段。
- **R5** 管理员可新增 / 编辑 / 停用会议室（名称、容量、位置、状态）。
- **R6** 每条预约记录保存预约人与预约时间。
- **R7** 会议室停用后不可再被预约；其已有预约保留、照常进行。
- **R8** 管理员可取消任意预约（不受「开始前」限制），释放该时段。

## 6. 非功能约束

- **并发**：两人同时预约同一会议室重叠时段，最多一人成功，不得双订（R3 须并发安全）。
- **安全/权限**：员工只能取消本人预约；预约人取自登录态。会议室维护仅 `admin`。
- **兼容性**：新增表，不改动既有数据。

## 7. 验收标准（AE，标注覆盖的 R）

- **AE1（R2/R3）** 两名员工同时预约「301 会议室 14:00–15:00」，仅一人成功，另一人收到「时段已被占用」。
- **AE2（R2）** 预约时结束时间 ≤ 开始时间，被拒并提示。
- **AE3（R4）** 开始前取消本人预约后，该时段重新可被预约。
- **AE4（R5/R1）** 管理员新增一间启用会议室后，员工列表立即可见、可约。
- **AE5（R7）** 停用某会议室后，员工不能再约它；它已有的预约仍在、不受影响。
- **AE6（R5）** 会议室名称超过 20 个字符时，新增/编辑被拒并提示「最多 20 个字符」（前端、后端、DB 三处一致）。
- **AE7（R8）** 管理员取消某员工的预约（即使已开始）后，该时段释放。

## 8. 覆盖矩阵

| 需求 | 覆盖它的验收 | 非功能约束 |
| --- | --- | --- |
| R1 查看会议室/占用 | AE4 | —— |
| R2 预约时段 | AE1、AE2 | 输入校验 |
| R3 重叠拒绝 | AE1 | **并发安全** |
| R4 取消 | AE3 | 仅本人 |
| R5 维护会议室 | AE4、AE6 | 仅 admin、名称长度 |
| R6 记录预约人/时间 | AE1（间接） | —— |
| R7 停用 | AE5 | —— |
| R8 管理员取消 | AE7 | 仅 admin |

> 每条 R 至少一条 AE；R6 由 AE1 成功路径间接验证。

## 9. 成功指标

- 会议室撞车投诉降为 0。
- 80% 的会议室预约通过系统自助完成（相对口头协调）。

## 10. 依赖与假设

- 复用现有用户与角色体系（`staff` / `admin`）。
- 假设会议时间在同一天内（跨天预约列为后续）。
