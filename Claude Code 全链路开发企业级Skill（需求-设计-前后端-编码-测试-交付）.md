# Claude Code 全链路开发企业级Skill（需求-设计-前后端-编码-测试-交付）

整套Skill为**公司统一全链路开发规范**，完全适配前文《Claude Code 企业级软件开发标准全流程》，覆盖从需求拆解、概要/详细设计、后端开发、前端开发、数据库SQL、代码审查、单元测试、Bug修复、文档交付全场景，所有AI输出严格遵循企业工程标准，实现全流程规范化、标准化、可落地。

使用说明：所有Skill可放入全局/项目局部Skill目录，自动加载生效，全程约束Claude Code输出，无需重复手动指定规范。

## 1. 00-enterprise-base.skill 企业通用全局基线（全流程必启）

**适用阶段**：需求、设计、编码、测试、交付、运维全阶段

**核心作用**：统一AI所有输出风格、安全、交付、沟通标准，是所有开发的底层约束

```yaml
name: Enterprise Full Base Standard
description: 对标字节跳动ByteStyle｜公司全流程开发通用基线，约束需求、设计、编码、测试、交付、运维所有AI输出，统一工程标准、安全标准、交付标准、协作标准
instructions: |
  # 字节风格 企业全局核心基线（全程强制执行、零例外）
  ## 一、核心价值观（对齐字节工程理念）
  1. 正确性优先：任何场景优先保证逻辑正确、数据正确、业务无事故，不追求快速落地
  2. 极简可维护：拒绝冗余设计、冗余代码、冗余逻辑，代码即文档，低心智负担
  3. 安全合规前置：安全、权限、数据脱敏、防风险为第一优先级
  4. 可观测可追溯：所有操作、变更、逻辑、异常可记录、可排查、可复盘
  5. 兼容稳定迭代：所有改动向前兼容，拒绝破坏性随意变更

  ## 二、AI 输出通用规范（严格对齐字节AI编码标准）
  1. 语言规范：全程中文沟通，技术术语、框架名词保留英文；输出结构化、分层化、结论前置，无废话、无铺垫、无模棱两可描述
  2. 思考透明：复杂需求必须先输出【执行方案】，明确改动文件、逻辑范围、风险点，人工确认后再落地
  3. 禁止幻觉：不编造不存在的接口、字段、方法、配置、业务规则，不确定内容主动标注待确认

  ## 三、安全合规规范（字节零事故标准）
  1. 严禁生成、泄露、硬编码任何密钥、AK/SK、数据库账号密码、内网IP、域名、敏感配置、业务隐私数据
  2. 所有敏感业务字段（手机号、身份证、地址、交易信息）必须默认脱敏展示，禁止明文返回、明文日志打印
  3. 禁止生成任何高危操作：清空全表、删除全量数据、关闭权限校验、绕过登录、绕过数据权限
  4. 所有接口、数据操作必须自带权限校验、防篡改、防重放、防越权兜底

  ## 四、工程交付规范（可上线、可审计、可运维）
  1. 所有输出的代码、脚本、配置、文档、SQL 必须：可直接运行、可直接上线、无语法错误、无逻辑漏洞
  2. 严格遵循「先设计、后编码；先方案、后改动」，禁止无依据、无规划直接编写/修改代码
  3. 所有改动遵循「最小改动原则」，不随意重构、不批量改无关代码、不破坏原有正常逻辑

  ## 五、代码质量基线（字节统一Code Review标准）
  1. 命名语义化：杜绝拼音、无意义缩写、a/b/c临时变量、中英文混合混乱命名
  2. 零魔法值：所有固定状态、类型、常量、阈值必须抽离枚举/常量类，代码中禁止硬编码
  3. 强健壮性：所有入参、外部返回值、集合对象必须做非空、边界、合法性校验，杜绝裸调用
  4. 单一职责：方法/类/接口只做一件事，单方法控制行数，逻辑分层清晰
  5. 资源必释放：流、文件、连接、线程、缓存资源必须主动关闭/释放，杜绝资源泄漏

  ## 六、注释与文档规范（代码即文档）
  1. 类、公共方法、核心接口必须标准文档注释：用途、入参、出参、异常、业务场景
  2. 复杂逻辑、特殊兼容、历史遗留逻辑、边界判断必须加行内注释，说明「为什么这么写」
  3. 注释简洁精准，不写废话注释、不重复代码逻辑、不保留废弃注释代码

  ## 七、文件与目录规范
  1. 所有文件、目录统一小写下划线命名，禁止中文、空格、大写、特殊字符
  2. 严格遵循项目现有目录结构，不擅自新增目录、不随意移动文件、不打乱工程分层
  3. 配置、脚本、静态资源、业务代码严格分类存放，不混放

  ## 八、异常与容错规范（线上高可用标准）
  1. 全链路异常捕获，禁止裸异常、禁止直接抛出原生异常
  2. 区分业务异常、系统异常、第三方异常，分层兜底、友好提示
  3. 关键流程必须有失败重试、降级、兜底策略，保证业务不崩、数据不脏

  ## 九、版本与Git协作规范（字节迭代标准）
  1. 开发采用小步增量提交，单一功能单一提交，提交信息规范清晰
  2. 功能提测、合入主干、上线前必须统一 squash 压缩为干净、可追溯的正式提交
  3. 禁止垃圾提交、无意义提交，保证Git历史干净可复盘、可回滚

  ## 十、性能基线（默认遵循）
  1. 禁止循环嵌套查询、循环创建实例、重复IO、无效遍历
  2. 批量场景优先批量处理，禁止逐条循环操作数据库/接口
  3. 查询优先走索引、分页、条件过滤，杜绝全表扫描、全量查询

```

## 2. 01-requirement-analysis.skill 需求拆解与澄清专项

**适用阶段**：阶段一 需求梳理、前期设计

**核心作用**：标准化需求拆解，杜绝需求遗漏、理解偏差、边界缺失

```yaml
name: Requirement Analysis Standard
description: 企业级需求拆解、澄清、结构化梳理规范
instructions: |
  # 需求拆解标准流程
  1. 接收原始需求后，先结构化拆解，输出：核心功能、业务目标、用户角色、使用场景
  2. 必须识别：正常场景、边界场景、异常场景、失败回滚场景
  3. 主动识别需求模糊点、矛盾点、缺失点，输出待确认问题清单
  4. 梳理模块依赖、功能前置条件、上下游系统关联关系
  5. 拆分需求为可落地的开发任务，匹配概要设计模块划分

  # 输出规范
  1. 输出结构化需求清单，条理清晰、无遗漏
  2. 输出风险点预判：技术风险、业务风险、兼容风险
  3. 输出对应概要设计填充内容，直接对接设计文档
```

## 3. 02-design-outline-detail.skill 概要+详细设计全流程规范

**适用阶段**：阶段一、阶段二 概要设计、详细设计

**核心作用**：严格对齐公司概要设计模板，统一图纸、表格、架构、流程输出标准

```yaml
name: Outline & Detail Design Standard
description: 企业级概要设计、详细设计全流程输出规范
instructions: |
  # 一、概要设计输出规则（宏观、架构、关系）
  1. 必须包含：系统架构、模块划分、角色权限、核心业务流程、ER图、接口规范、非功能需求、风险方案
  2. 只做整体设计、不写代码细节、不贴完整字段、不写具体接口入参出参

  3. 接口：细化全部入参、出参、校验规则、错误码、请求示例、返回示例
  4. 代码结构：明确分层结构、公共组件、工具类、枚举、异常体系

  # 三、设计通用约束
  1. 所有设计内容贴合业务，不设计冗余、无用模块
  2. 兼顾扩展性，预留后续迭代、渠道、字段扩展能力
  3. 所有设计文档可直接评审、可直接作为开发依据、可直接归档
```

## 4. 03-backend-enterprise-dev.skill 后端全栈开发规范（Java/Go通用）

**适用阶段**：阶段三 编码开发、接口开发、业务逻辑开发

**核心作用**：统一后端分层、编码、事务、日志、异常、调用规范

```yaml
name: Backend Enterprise Dev Standard
description: 企业级后端开发通用规范，适配Java SpringBoot、Go等主流技术栈
instructions: |
  # 一、架构分层约束
  1. 严格遵循分层架构：Controller → Service → Mapper/Repository → Entity
  2. 严格分层调用，禁止跨层直连、禁止Controller直接操作数据库
  3. 禁止数据库Entity直接对外返回，必须使用DTO/VO数据转换

  # 二、编码通用规范
  1. 语义化命名，禁止拼音、无意义缩写、魔法值、硬编码
  2. 固定常量、状态值、类型值全部抽离枚举/常量类
  3. 单函数单一职责，代码简洁可维护，拒绝冗余炫技代码
  4. 所有入参必校验：非空、范围、格式、业务合法性

  # 三、业务容错规范
  1. 数据库多表操作、多步骤业务必须加事务，保证原子性
  2. 外部接口调用必须加超时、重试、熔断、降级兜底
  3. 文件、流、连接、线程等资源必须主动关闭释放
  4. 全局统一异常处理，不对外暴露原生异常堆栈

  # 四、日志规范
  1. 关键链路打印入参、出参、耗时、异常信息
  2. 日志分级输出：INFO正常、WARN预警、ERROR异常
  3. 日志禁止打印手机号、密码、身份证等敏感数据
```

## 5. 04-sql-db-enterprise.skill 数据库SQL专项规范

**适用阶段**：设计阶段、编码阶段、优化阶段

**核心作用**：杜绝慢SQL、烂表结构、索引问题，保障数据安全与性能

```yaml
name: SQL & Database Enterprise Standard
description: 企业级数据库设计、建表、SQL编写、性能优化规范
instructions: |
  # 一、建表规范
  1. 表名、字段名小写下划线，语义化清晰
  2. 所有数据表必须包含：主键id、创建时间、更新时间、逻辑删除字段
  3. 字段设置合理类型、长度、非空、默认值，减少NULL值
  4. 禁止大字段滥用，长文本、大文本单独分表存储

  # 二、索引规范
  1. 基于查询、筛选、排序场景建索引，遵循最左匹配原则
  2. 禁止过度索引、无效索引
  3. 唯一业务字段建立唯一索引，防止重复数据

  # 三、SQL编写规范
  1. 禁止 select *，只查询业务所需字段
  2. 禁止全表更新、全表删除，操作必须带有效where条件
  3. 禁止嵌套多层子查询，优先使用join关联
  4. 批量操作使用批量SQL，禁止循环单条操作
  5. 禁止大事务、长事务，拆分复杂事务

  # 四、数据安全规范
  1. 线上SQL必须先校验、后执行
  2. 敏感数据字段设计脱敏存储、查询脱敏返回
```

**适用阶段**：前后端并行开发、页面开发、组件开发

**核心作用**：统一前端组件、状态、请求、样式、性能规范

```yaml
name: Frontend Vue/React Enterprise Standard
description: 企业级前端开发规范，适配Vue2/Vue3/React全场景
instructions: |
```

## 7. 06-code-review-optimize.skill 代码审查与重构优化

**适用阶段**：阶段四 代码审查、优化重构

**核心作用**：自动化企业代码门禁，提前拦截Bug、漏洞、性能问题

```yaml
name: Code Review & Optimize Standard
description: 企业级代码审查、性能优化、安全修复、重构规范
instructions: |
  # 审查核心维度（全覆盖）
  1. 规范性审查：命名、注释、分层、代码风格、魔法值、硬编码
  2. 安全审查：SQL注入、XSS、权限绕过、敏感数据泄露、接口安全
  3. 健壮性审查：空指针、参数未校验、异常未捕获、资源未释放
  4. 性能审查：循环嵌套、重复查询、慢SQL、无效循环、内存浪费
  5. 可维护性审查：代码冗余、逻辑耦合、职责不单一、无法复用

  # 优化修复规则
  1. 先输出完整问题报告，标注风险等级、问题位置、危害说明
  2. 针对性给出优化方案，不盲目改动原有正确逻辑
  3. 重构遵循最小改动原则，保证功能不变、逻辑兼容
  4. 优化后再次校验，确保无新增Bug、性能提升、规范合规
```

## 8. 07-test-unit-bugfix.skill 单元测试与Bug修复专项

**适用阶段**：阶段五 测试、Bug修复、联调

**核心作用**：标准化测试用例生成、Bug定位、修复、回归闭环

```yaml
name: Unit Test & Bug Fix Standard
description: 企业级单元测试生成、问题排查、Bug修复闭环规范
instructions: |
  # 一、单元测试规范
  1. 针对核心业务逻辑生成测试用例，覆盖正常、边界、异常场景
  2. 测试用例可直接运行、断言精准、覆盖核心分支
  3. 优先覆盖业务核心流程、高频接口、复杂计算逻辑

  # 二、问题排查规范
  1. 接收报错后，优先分析日志、堆栈、调用链路，定位真实根因
  2. 区分：代码Bug、配置问题、环境问题、第三方依赖问题
  3. 输出完整排查过程、根因结论、影响范围

  # 三、Bug修复规范
  1. 给出两套方案：临时应急方案（快速止血）+ 永久根治方案
  2. 修复代码遵循原有架构与规范，不破坏整体逻辑
  3. 修复后输出回归测试要点，确保问题彻底解决，无次生问题
  4. 同类问题给出预防方案，避免重复踩坑
```

## 10. 全链路Skill加载顺序（团队标准配置）
